0. Read First — How To Build (client instructions)

These are the binding instructions for this portal. Everything below expands on them.

  1. Separate web app, outside the mobile app. A new folder infra-website sits beside the React-Native app (infra/). It is the admin control panel.
  2. Stack: React + Vite + JavaScript (JSX), NOT TypeScript. Styling with Tailwind CSS. Keep it clean and stylish.
  3. One backend only. Reuse the existing Express server at infra/server (same MySQL DB the app uses). Add /api/admin/* routes there — do not spin up a second backend.
  4. Everything dynamic. Whatever the app shows — services, food menus, doctors, cars/autos, housekeeping, wheelchair, temples, community, rewards, etc. — must be created/edited/removed by the admin. The app reads exactly what the admin writes.
  5. Images via Cloudinary with a drag-&-drop uploader and an upload-by-link (paste URL) option, reused everywhere an image is needed.
  6. Full user-management portal inside the admin (residents, KYC, bookings, status).
  7. Module → Phase structure. This PDF lists modules; each has phases (large sub-tasks). Build one module at a time, end-to-end, then the next.
  8. Track progress in contextadmin.md — every module + the one currently in progress — so work can resume after closing.
Golden rule: if the app screen shows it, the admin must be able to manage it. This document maps every app surface to an admin module.

PART A — Foundations

1. Purpose, scope & principles

The Yamuna Infra resident mobile app (React Native) today ships with hard-coded / mock content for every service. This portal replaces that with a fully admin-managed backend so non-technical staff can run the whole platform: add a new cook, change a dosa's price and photo, add a temple, define a cab fare, broadcast a notice, approve a KYC, etc.

Core principles

  • Dynamic-first: every list/card/price/image in the app comes from the database, written by the admin. No app rebuild needed to change content.
  • One backend, one DB: the portal and the app talk to the same Express server. Admin writes; app reads.
  • CRUD parity: for each app surface there is a matching admin screen with Create / Read / Update / Delete + enable/disable + ordering.
  • Image-rich: Cloudinary-backed images on every catalog entity (providers, food, doctors, temples, projects, amenities…).
  • Role-based: superadmin / manager / support with scoped permissions.
  • Incremental: ship module by module; never a big-bang.

5. Data model overview

Most catalog tables follow a common shape so a generic CRUD backend + a generic DataTable frontend can serve many modules. Add an image_url column (Cloudinary) and is_active, sort_order to every catalog table.

Common columns (every catalog entity)

ColumnTypePurpose
idBIGINT PKIdentity
name / titleVARCHARDisplay
descriptionTEXTDetails
image_urlVARCHARCloudinary secure_url
price / amountINTWhere applicable (₹)
is_activeTINYINTShow/hide in app
sort_orderINTDisplay order
created_at / updated_atDATETIMEAudit

Key entity groups

  • Catalog: service_categories, service_providers, provider_offerings, food_categories, food_items, doctors, specialties, mobility_equipment, wellness_services, spiritual_services, temples, vehicle_types, transport_places, amenity_categories, amenities, reward_offers, projects.
  • Transactional: bookings (per service), food_orders, appointments, rides, amenity_bookings, visitor_passes, sos_requests, payments, redemptions, referrals, reminders.
  • People: users (residents), family_members, kyc, emergency_contacts, admins, roles.
  • Content/Config: announcements, events, daily_content, notifications, app_settings, reminder_categories, ai_knowledge_sources.
Several of these tables already exist for the app (Modules 1–36). The admin work mostly adds write endpoints + image_url/is_active/sort_order columns + admin UIs on top of existing read endpoints.

6. Authentication, roles & security

  • Admins are separate from residents. New admins table (email, password_hash, role, name, is_active). Resident OTP login is untouched.
  • Login: email + password → bcrypt verify → JWT ({ sub, role, type:'admin' }, 12h). Optional OTP/2FA later.
  • Roles: superadmin (everything incl. admin/role mgmt), manager (all content + users + orders), support (read + limited actions: KYC review, order status, notifications).
  • Middleware: requireAdmin(roles?) verifies the JWT, checks type==='admin' and role membership.
  • Frontend guard: a <Protected/> wrapper redirects to /login when no token; hides nav items the role can't access.
  • Security: rate-limit login, helmet, CORS allowlist (portal origin), validate every body with Joi (already used), never expose service keys to the browser (Cashfree/NVIDIA stay server-side; only Cloudinary unsigned preset is public).

7. Media handling — Cloudinary

  • Component: <ImageUploader value onChange /> used by every image field. Two ways to set an image:
    1. Drag & drop (or click to pick) a file → uploads to Cloudinary → stores secure_url.
    2. Paste a link → validates & stores the URL (optionally re-hosted via Cloudinary fetch).
  • Unsigned preset for browser uploads (VITE_CLOUDINARY_CLOUD_NAME + VITE_CLOUDINARY_UPLOAD_PRESET). For stricter control add a backend /api/admin/media/sign route (signed uploads).
  • Media Library (A17): browse/search past uploads, organise in folders, reuse via a picker.
  • Transforms: store base secure_url; the app/portal request sized variants via Cloudinary URL params (e.g. w_400,q_auto,f_auto).

PART C — Process & Delivery

9. Delivery process & contextadmin.md

We build one module at a time, end-to-end, exactly like the app was built. For each module:

  1. Backend migration (new tables / columns like image_url).
  2. Backend model + controller + routes under /api/admin (+ Joi validation).
  3. Quick backend smoke test (curl / REST client).
  4. Frontend pages + wire to API using the reusable components.
  5. Manual test against the running app (admin writes → app shows).
  6. Mark the module/phase done in contextadmin.md.
contextadmin.md lives in infra-website/ and mirrors the app's CONTEXT.md: a module-status table + a "current focus" section + per-module notes. Always update it at the end of a work session so the next session resumes cleanly.

Suggested build order

A0 (done) → A1 (auth) → A0.3–0.5 (components + uploader) → A3 (users) → A4 (services) → A5 (food) → A9 (temples) → A10 (transport) → A11 (amenities) → A6 (healthcare) → A8 (wellness/spiritual) → A7 (mobility) → A12 (community) → A13 (rewards/projects) → A14 (notifications) → A2 (dashboard) → A16 (payments) → A17 (media) → A18 (settings) → A15 (AI/RAG) → A19 (audit).

10. Deployment

  • Backend: already on the shared host; add admin routes + run new migrations. Set Cloudinary + admin secrets in server/.env.
  • Admin portal: npm run build → static dist/. Host on Netlify/Vercel/any static host, or serve via the Express server (express.static). Lock CORS to the portal origin.
  • Env: VITE_API_BASE_URL (prod API), VITE_CLOUDINARY_*. Never put server secrets in the Vite build.
  • Auth: HTTPS only; short-lived admin JWT; optional IP allowlist for the admin origin.
